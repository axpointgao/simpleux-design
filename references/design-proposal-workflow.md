# Design Proposal Workflow：设计提案

设计提案用于项目早期设计定义或方向评审。方向来自用户输入或用户确认；simpleux-design 负责把方向组织成清晰、专业、可比较、可汇报的提案，不替用户决定方向。

## 工作目标

- 整理用户提供或确认的多个方向。
- 识别方向差异发生在视觉层、交互层，还是交互与视觉组合层。
- 组织方向总览、方向表达、差异对比、评审关注点和后续详细设计衔接。

## 方向输入不足

如果当前只有单一方向，不切换到其他场景。先帮助用户补齐设计提案输入：

- 询问是否有其他方向的描述、草图、参考、关键词、样稿或内部讨论记录。
- 可基于现有项目目标、调研洞察、设计策略和参考提出 2-3 个方向建议草案。
- 方向建议草案必须经用户确认、修改或否定后才能成为正式输入。

## 差异层识别

### 视觉多方向

用户提供一套交互方案和两个以上视觉方向。

- 明确统一交互基准。
- 重点比较视觉主张、品牌表达、组件语言、信息密度、情绪关键词和关键页面观感。
- 不硬编交互差异。

### 交互多方向

用户提供一套视觉语言和两个以上交互方向。

- 明确统一视觉基准。
- 重点比较信息架构、首页组织、核心路径、模块组织、反馈机制和可扩展性。
- 不硬编视觉差异。

### 完整多方向

用户提供两个以上同时包含交互和视觉差异的方向。

- 组织每个方向的交互主张、视觉主张、关键页面、样稿状态和综合对比。
- 只比较用户输入或确认的真实差异。

## 推荐结构

```text
项目输入与设计目标
→ 方向总览
→ 方向 A 表达
→ 方向 B 表达
→ 方向 C 表达
→ 方向对比
→ 评审关注点
→ 后续详细设计衔接
```

每个方向可包含：

- 方向名称
- 核心主张
- 设计关键词
- 交互层表达
- 视觉层表达
- 关键页面/流程
- 用户体验价值
- 业务/品牌/运营价值
- 样稿展示或占位说明
- 后续深化注意事项

## 样稿状态

- **已有正式样稿**：展示、说明、提炼亮点。
- **未提供正式样稿**：使用线框示意、结构占位、典型页面说明或待替换样稿区，并说明后续可替换。

## 质量检查

- 是否忠实组织用户提供或确认的方向。
- 是否明确共用基准和差异变量。
- 是否避免擅自生成新方向、替用户选择方向、硬编不存在的交互或视觉差异。
- 是否说明各方向对业务目标、品牌目标或运营目标的支撑差异。
