# Project Solution Workflow：项目方案

项目方案是用户体验设计咨询场景，不是销售论证。客户进入委托或咨询阶段时，通常已经明确项目必要性；SimpleUX 的价值是理解项目处境，把客户诉求转译为 UX 问题，并提出专业判断、工作路径、交付边界和协作机制。

## 工作目标

输出一份能够说明“这个项目如何做对”的方案：

- 项目理解
- 体验问题界定
- 设计判断
- 策略建议
- 工作方案
- 交付与协作
- 业务价值连接

## 推荐流程

1. **项目理解**：复述并深化客户已提出的课题、目标、边界、决策人关注点和项目语境。
2. **现状与资料判断**：列出已有资料、截图、竞品、访谈或业务输入，区分明确事实、推断和缺失证据。
3. **体验问题界定**：把客户语言转译为 UX 问题。
4. **设计判断**：判断项目重心、根因与表象、优先处理页面/流程、体验/业务/品牌平衡和应避免误区。
5. **策略建议**：提出服务项目推进的体验策略。
6. **工作方案**：说明研究诊断、问题确认、信息架构、核心流程、关键页面、视觉方向、设计系统、评审协同和迭代。
7. **交付与协作**：明确阶段交付物、最终交付物、客户确认节点、客户输入要求、范围边界和后续衔接。

## 问题界定维度

把“App 不好用、页面老、用户不活跃、转化差、视觉不精致”转成可设计的问题：

- 信息架构
- 核心任务路径
- 内容理解
- 信任与安全感
- 品牌感知
- 运营触点
- 关键转化链路
- 设计系统一致性

不要只复述客户原始诉求。

## 外部研究指引

缺少资料时，给用户可交给 Gemini、Deep Research 或人工研究的 brief。研究 brief 应包含：

- 研究目标
- 研究对象：客户、行业、竞品、用户、品牌、现有产品、业务指标
- 重点问题
- 希望返回的结构化输出
- 需要截图、来源链接或数据支撑的结论

银行 App 项目可提示研究银行类型、排名、资产规模、服务区域、品牌定位、发展阶段、主力客群、业务 KPI、现有 App 体验问题、竞品体验策略和评审关键人关注点。

## 业务价值表达

所有关键策略都要形成：

```text
设计动作 → 用户/运营行为变化 → 业务价值
```

缺少真实数据时，用“支撑、促进、降低风险、提高机会”等谨慎表述，不伪造量化收益。

## 质量检查

- 是否避免销售论证和“为什么要做”的推销口吻。
- 是否体现项目理解、问题界定、设计判断、工作路径、交付物和协作边界。
- 是否避免空泛价值宣称和无证据结论。
- 是否将体验设计工作连接到业务目标或项目成效。
