# POC Proposal Workflow：POC 提案

POC 提案是招标或投标场景中的设计命题作文。它负责组织投标材料中的 POC 设计方案表达，不生成完整投标章节，不覆盖项目规划、实施计划、团队能力、风控质保或商务响应。

## 工作目标

让评审者快速理解：

- 命题是什么
- 评分标准关注什么
- SimpleUX 如何理解体验问题
- 设计响应如何覆盖评分点
- 样稿或占位表达什么价值
- 后续如何深化

## 样稿状态

只分两类：

- **已有正式样稿**：分析样稿、组织说明、提炼亮点和生成展示页。
- **未提供正式样稿**：使用线框示意、结构占位、待替换样稿区或样稿表达要求，并标注为示意/占位。

不要把未提供正式样稿细分为“并行样稿”或“无样稿”。在 PPT 制作当下，两者都没有可直接放入 PPT 的正式样稿。

## 推荐流程

1. **命题拆解**：拆出命题对象、用户对象、业务目标、设计要求、限制条件和交付形式。
2. **评分标准映射**：把评分项转译为设计表达任务。
3. **体验问题界定**：指出该命题背后的 2-4 个关键 UX 问题。
4. **设计响应策略**：提出短、准、能落到样稿或占位的响应策略。
5. **样稿表达规划**：说明正式样稿、线框或占位要表达的页面目标、模块、信息层级、交互逻辑和视觉重点。
6. **价值回应**：连接用户价值、业务价值、品牌价值、评分价值和落地价值。
7. **后续深化**：说明正式样稿替换或详细设计深化方向。

## 设计方案表达结构

可使用：

```text
命题理解
→ 评分标准拆解与响应策略
→ 关键体验问题界定
→ 设计响应策略
→ 样稿展示或线框占位说明
→ 关键交互与视觉亮点
→ 价值回应与后续深化
```

每个关键设计点都要映射到对应设计要求或评分关注点。

## 禁止事项

- 不扩展为完整投标章节生成工具。
- 不写实施计划、团队能力、风控质保或商务响应。
- 不把线框、结构占位或待替换样稿区伪装成正式样稿。
- 不伪造评分标准、业务数据或量化收益。

## 质量检查

- 是否回应 POC 命题。
- 是否映射评分标准。
- 是否说明样稿状态。
- 是否讲清设计响应和业务/评分价值。
- 是否避免非设计章节扩写。
