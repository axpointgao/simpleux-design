# Logic Graphics：逻辑图形组件库

本文件用于模型图、关系图、路径图、系统图、流程图、矩阵图、信息图和图解页等结构化图形。图形只是增强理解的手段，目标是提供可参考的组件、示例、模板母题和质检方法，而不是要求每个 deck 页面都使用组件库，也不是复刻外部图库或历史页面。需要更强表现力时，可从当前维护的模板母题中选择结构，再按内容替换与 token 适配。

## 使用时机

遇到以下内容，必须读取本文件并做逻辑图形选型；适合结构化表达、复用或硬门禁质检时，应使用 `assets/logic-graphics/` 组件库或模板母题：

- 概念模型、思维模型、能力模型。
- 节点关系、因果关系、依赖关系。
- 路径、流程、阶段、闭环。
- 系统模块、业务架构、矩阵分类。
- 信息图、图解页或需要在 deck 页面内放置稳定可读的图形区域。

组件库不是所有页面的强制路径，但“选型判断”是强制路径。先用设计判断决定是否需要图形：如果图形不能让关系、方向、层级或转化更清楚，就保持常规版式，并说明或记录原因。简单的局部图解可以直接使用当前 deck 的 HTML/CSS；当图形需要复用组件、自动布局、模板母题或硬门禁质检时，必须把内容转成“组件 + 关系”的 JSON，或套用 P1-P8 母题生成 HTML/SVG。避免临场手写难以维护的复杂自由 SVG。

如果页面标题或内容包含“模型、关系、流程、路径、系统、矩阵、闭环、架构、能力地图、服务地图、体验链路、经营逻辑、价值链路”等表达，默认视为需要逻辑图形选型。除非普通文字/分区明显更清楚，否则不要完全跳过组件库或模板母题。

## 设计原则

### 亲密性

相关节点必须靠近并形成组，不相关节点必须拉开距离。属于同一主题的节点优先放进同一 `GraphicGroup`，或通过相同 `group` 字段进入同一布局区。不同区块之间用留白区隔，不用更多边框堆叠。

### 对齐

节点、标签、旁注和分组必须沿边缘、中心轴或网格轨道对齐。不要临场随意放置元素。路径图沿主轴对齐，分层图沿行列对齐，矩阵图沿网格对齐，中心扩散图围绕中心节点对齐。

### 重复

同一层级的组件必须共享 token。标题字号、正文字号、连接线粗细、圆角、边框、阴影、节点内边距和组间距不得临时变化。需要强调时使用 `emphasis` 或 `tone`，不要手写局部样式。

### 对比

每页最多 1 个主焦点。主焦点可以更大、更深、更亮或使用强调色；次要节点必须降级。对比用于突出表达重点，不用于装饰。若有多个重点，应拆成多页或改成模块系统。

### 留白

图形必须留出安全区和区块间距。主图形区不压标题、副标题、页脚或出品方署名。节点之间留出连接线和关系标签的空间，避免箭头贴着文字穿过。

### 层级

页面层级建议为：标题区、核心图形、关系说明、旁注、页脚。不要让旁注或连接标签抢过核心节点。组件库会将标题区、主图形区和页脚区分开，生成后必须跑质检。

## 绘制质量门槛

图形页不是“节点加箭头”就完成。无论使用 `simple-custom`、基础组合示例还是模板母题，交付前都必须用以下标准做轻量自检：

- **不遮挡**：文字、节点、徽标、连接标签、箭头和装饰线不得互相覆盖；中心标题和关键数字周围必须留出呼吸空间。
- **不穿字**：连接线不得穿过正文、标题、编号和节点内部文本；如果路径不可避免地穿过信息区，应改用分区、编号路径或方向带。
- **少箭头**：箭头只用于真实方向、因果、流转或反馈关系；并列、归类和弱关联优先用空间位置、组块、色带或编号表达。
- **箭头自然**：曲线箭头的终点切线必须与箭头朝向一致；若末端角度不自然，优先改直线、折线、短连接线或拆成两个关系。
- **边缘留距**：箭头尖端与节点、卡片和圆形边缘保持可见间距，不插入形状内部，不贴近文字基线。
- **避免透明穿帮**：关键连接线和箭头优先使用实色浅线或低对比实色；不要让半透明 marker 叠在线尾形成脏色、断裂或双影。
- **降低复杂度**：当连线超过理解负担，优先减少箭头、拆页或改为三栏流程、矩阵、分层模块，而不是继续微调复杂路径。

这些规则同样适用于普通 HTML/CSS 自定义图解。带 `data-logic-graphic` 的组件库产物仍需运行硬门禁质检；不带该标记的 `simple-custom` 图解也应人工检查上述门槛。

## 组件边界

- `GraphicFrame`：固定 1920×1080 画布、安全区、标题区、主图形区、页脚区。
- `GraphicNode`：圆形、矩形、胶囊、菱形、编号点，承载标题和短说明。
- `GraphicLabel`：标题、说明、注释、数字、短标签，优先使用 HTML 文本块。
- `GraphicConnector`：直线、折线、曲线、箭头、虚线，只表达关系，不承载长句。
- `GraphicGroup`：相关节点的背景容器或模块区块。
- `GraphicBadge`：阶段、状态、关键词，必须短。
- `GraphicCallout`：重点解释、旁注、风险提示，不能替代正文段落。

## 输入接口

```json
{
  "canvas": { "title": "页面主结论", "subtitle": "可选说明", "theme": "simpleux-light" },
  "layout": { "type": "stack | path | radial | matrix | loop | modules" },
  "nodes": [
    { "id": "n1", "kind": "circle | rect | pill | diamond | index", "title": "节点标题", "body": "短说明", "group": "可选分组", "emphasis": true }
  ],
  "connectors": [
    { "from": "n1", "to": "n2", "kind": "line | elbow | curve | arrow", "label": "可选关系" }
  ],
  "callouts": [
    { "target": "n1", "title": "提示标题", "body": "提示说明" }
  ]
}
```

## 容量规则

- 标题区：主标题建议 14 到 30 个中文字符，副标题建议 40 个中文字符以内。
- 节点标题：建议 4 到 12 个中文字符，最长不超过 18 个中文字符。
- 节点说明：建议 12 到 32 个中文字符，最长不超过 48 个中文字符。
- 连接标签：最长不超过 12 个中文字符。
- 徽标：最长不超过 8 个中文字符。
- 旁注：标题最长 14 个中文字符，正文最长 56 个中文字符。
- 当信息超出容量时，优先拆节点、拆页或改为模块系统，不压缩字号硬塞。

## 图形选型

| 内容关系 | 可考虑的表达 | 说明 |
| --- | --- | --- |
| 轻量概念解释、少量标签关系 | `simple-custom` 自定义 HTML/CSS 图解 | 不必读取或调用组件库，保持当前 deck 版式一致。 |
| 单中心、路径、分层、矩阵、闭环、模块等低复杂度结构 | 基础组合示例 + 自主改写 | 示例用于学习 JSON、布局和组件调用方式，不是正式可直接交付模板。 |
| 复杂关系、强视觉识别或接近模板母题的页面 | P1-P8 模板母题 | 按当前内容替换、删减、拆页和 token 适配，不逐像素复刻。 |
| 超出容量、多个主焦点或长文本说明 | 拆页、改普通 deck 版式或改自定义图解 | 不压缩字号硬塞，不为了套模板牺牲可读性。 |

## 基础组合示例

基础组合示例用于说明低复杂度结构如何用 JSON、布局字段和组件组合表达。它们不是正式可直接放入 deck 的稳定结构，也不是高表现力母题；真实项目需要根据当前页内容、品牌、版式和容量重新组织。

- `radial` / 单中心关系示例：一个核心概念与周边支撑条件。
- `path` / 横向阶段路径示例：项目推进、交付路线、实施步骤。
- `stack` / 上下分层结构示例：目标层、机制层、执行层等纵向递进关系。
- `matrix` / 四象限矩阵示例：优先级、风险收益、投入产出等二维判断。
- `loop` / 环形闭环示例：增长飞轮、运营循环、学习闭环。
- `modules` / 模块架构示例：平台架构、业务能力地图、组织协作分工。

`component-gallery` 只作为开发调试入口，用于检查底层组件渲染能力，不作为正式组合推荐给用户。

当内容只是通用关系、阶段、矩阵、闭环或模块架构时，可以参考基础组合示例并自主改写；当表达需要更强视觉识别、复杂结构或参考图风格时，再选择模板母题或自定义图解。

## 模板母题

模板母题是少量高质量结构参考，不是整页坐标复刻。每个母题必须支持内容替换、删减、拆页、风格 token 替换和质检；如果当前内容不适合母题容量或关系结构，优先换表达方式，不要为了套母题强行塞入。

正式母版只保留 8 个，并按 P1 到 P8 连续编号：P1 三角循环能力、P2 双侧聚焦对照、P3 中心轨道网络、P4 阶段通道推进、P5 三元环绕概念、P6 分层核心诊断、P7 部署轨道步骤、P8 双端核心闭环。

### P1 三角循环能力

适用场景：表达一个三层能力内核、两侧动作路径和底部反馈闭环。适合“输入-处理-输出-反馈”“认知-转化-协同”等结构。

输入字段：

```json
{
  "type": "triangle-cycle",
  "canvas": { "title": "", "theme": "takram" },
  "topLabel": "设计功能",
  "topBullets": ["顶部短要点", "顶部短要点"],
  "sideLabels": { "left": "设计功能", "right": "设计功能" },
  "levels": [
    { "title": "洞察层" },
    { "title": "转化层" },
    { "title": "协同层" }
  ],
  "callouts": [
    { "slot": "left", "title": "输入侧", "lines": ["短句", "短句", "短句"] },
    { "slot": "right", "title": "输出侧", "lines": ["短句", "短句", "短句"] },
    { "slot": "bottom", "title": "反馈侧", "lines": ["短句", "短句"] }
  ]
}
```

容量边界：

- `levels` 稳定支持 2 到 4 层，推荐 3 层。
- 每层标题最长 14 个中文字符，推荐 4 到 8 个字。
- 顶部要点最多 4 条，总长度不超过 90 个中文字符。
- 三张说明卡固定为 `left`、`right`、`bottom`，每张最多 1 个标题和 3 条短句；底部卡建议 2 条以内。
- 主题适配优先使用 `canvas.theme` 或 `style`，再由母题映射到语义 token；只有品牌强约束时才用 `themeTokens` 覆盖。
- 母题内部坐标只表达结构关系，不为某个风格单独改坐标。色彩、投影、圆角、卡片边框和环线颜色必须通过 token 改。
- 嵌入 PPT 时必须把 1920×1080 图形作为整体等比缩放。不要拆开内部 SVG、HTML 文本和卡片单独拉伸，否则文字与形状会失去坐标关系。
- 若缩放后宽度低于 960px，优先减少说明卡文本或拆页；不要继续压缩字号。

### P2 双侧聚焦对照

适用场景：表达两个核心体验焦点、左右两组目标或约束，以及中间的业务/客户/用户关系层级。适合“视觉 vs 交互”“产品侧 vs 用户侧”“前台体验 vs 后台能力”等对照结构。

输入字段：

```json
{
  "type": "dual-focus-cone",
  "canvas": { "title": "", "theme": "takram" },
  "left": {
    "title": "视觉",
    "subtitle": "Vision",
    "points": ["满足用户视觉偏好", "满足产品体验统一性", "实现商业上的成功"]
  },
  "right": {
    "title": "交互",
    "subtitle": "Interaction",
    "points": ["交互反馈一致性", "保持简洁的流畅性", "玩法简单无需思考"]
  },
  "axis": {
    "top": "业务侧",
    "middle": "客户侧",
    "bottom": "用户侧"
  }
}
```

容量边界：

- 左右焦点固定为两个圆形，`title` 最长 8 个中文字符，`subtitle` 最长 16 个英文字符。
- 左右要点各稳定支持 1 到 3 条，推荐 3 条；每条最长 18 个中文字符。
- 中央轴固定为 3 个标签，标签最长 8 个中文字符。
- 左右投射三角只表达“聚焦/投射关系”，不承载文字，不参与自由变形。
- 若某侧要点超过 3 条，优先拆页或改用模块系统，不继续压缩字号。

### P3 中心轨道网络

适用场景：表达一个起点或来源，经由中心枢纽连接若干轨道要素，再扩散到一组外部结果节点。适合“中心平台 + 周边因素 + 外部参与者/结果”“核心机制 + 反馈网络”“源头-枢纽-目标群”的结构。

输入字段：

```json
{
  "type": "hub-orbit-network",
  "canvas": { "title": "", "theme": "takram" },
  "source": { "title": "发起源" },
  "hub": { "title": "核心", "subtitle": "Service" },
  "orbitNodes": [
    { "title": "用" },
    { "title": "想" },
    { "title": "拼" },
    { "title": "比" }
  ],
  "orbitLabels": {
    "top": "价值度",
    "left": "品质感",
    "right": "效益比",
    "bottom": "信任感",
    "outerTop": "品牌传递",
    "outerBottom": "价值认同"
  },
  "targets": [
    { "title": "参与者A" },
    { "title": "参与者B" },
    { "title": "参与者C" }
  ]
}
```

容量边界：

- `source` 固定为 1 个起点节点，标题最长 10 个中文字符。
- `hub` 固定为 1 个中心节点，标题最长 8 个中文字符，副标题最长 16 个英文字符。
- `orbitNodes` 稳定支持 4 个短节点，每个最长 4 个中文字符。
- `orbitLabels` 固定为内圈 4 个关系标签和外圈 2 个关系标签，每个最长 10 个中文字符。
- `targets` 稳定支持 2 到 3 个结果节点，推荐 3 个，每个最长 10 个中文字符。
- 左右推进箭头只表达方向，不承载文字；内外虚线椭圆只表达关系路径，不作为内容容器。

### P4 阶段通道推进

适用场景：表达从起点进入一条阶段通道，经由 2 到 3 个中间阶段推进到终点，并在两端挂载外部输入、校准或约束标签。适合“基础准备-阶段处理-结果沉淀”“输入调研-能力加工-输出验证”等路径结构。

输入字段：

```json
{
  "type": "ribbon-stage-pipeline",
  "canvas": { "title": "", "theme": "takram" },
  "start": {
    "title": "起点\n基础",
    "badges": [
      { "title": "输入\n调研", "position": "top" },
      { "title": "外部\n约束", "position": "right" },
      { "title": "反馈\n校准", "position": "bottom" }
    ]
  },
  "stages": [
    { "title": "阶段一", "lines": ["识别关键要素", "建立判断标准"], "guide": "top" },
    { "title": "阶段二", "lines": ["组合表达路径", "沉淀执行动作"], "guide": "top" }
  ],
  "end": {
    "title": "终点\n结果",
    "badges": [
      { "title": "验证\n反馈", "position": "top-left" },
      { "title": "持续\n迭代", "position": "bottom-left" }
    ]
  }
}
```

容量边界：

- `start` 和 `end` 固定为两端主节点，标题最长 10 个中文字符，推荐 2 行短词。
- `stages` 稳定支持 2 到 3 个阶段，推荐 2 个；每个阶段标题最长 10 个中文字符，说明最多 2 行，总长度不超过 42 个中文字符。
- `start.badges` 最多 3 个，`end.badges` 最多 2 个，每个徽标最长 8 个中文字符。
- 透视带只表达推进通道，不承载文字；虚线弧只表达端点与辅助标签的关联，不压入端点正文。
- 若阶段超过 3 个，优先拆为路径图或模块系统；不要把阶段圆继续缩小硬塞。

### P5 三元环绕概念

适用场景：表达一个中心概念由三个关键构成要素共同支撑，并通过环形箭头形成循环强化关系。两侧属性胶囊用于承载人群、调性、条件或约束，不参与主循环。适合“中心心智 + 三个视觉/体验要素 + 外围属性”“核心主张 + 三类驱动因素 + 标签组”等结构。

输入字段：

```json
{
  "type": "triad-orbit-concept",
  "canvas": { "title": "", "theme": "takram" },
  "center": {
    "title": "核心概念\n信息传达",
    "subtitle": "Central concept"
  },
  "labels": { "left": "关系标签" },
  "nodes": {
    "top": { "title": "构成要素", "subtitle": "短说明" },
    "left": { "title": "构成要素", "subtitle": "短说明" },
    "right": { "title": "构成要素", "subtitle": "短说明" }
  },
  "pillGroups": {
    "left": [{ "title": "标签" }, { "title": "标签" }, { "title": "标签" }],
    "topRight": [{ "title": "标签" }, { "title": "标签" }],
    "right": [{ "title": "标签" }, { "title": "标签" }, { "title": "标签" }, { "title": "标签" }]
  }
}
```

容量边界：

- `center.title` 最长 14 个中文字符，推荐 2 行；`center.subtitle` 最长 24 个英文字符。
- 三个环绕节点固定为 `top`、`left`、`right`，每个标题最长 10 个中文字符，副标题最长 18 个中文字符。
- `pillGroups.left` 稳定支持 3 个胶囊，`pillGroups.topRight` 支持 2 个，`pillGroups.right` 支持 4 个；每个胶囊最长 8 个中文字符。
- 环形箭头只表达循环方向，不承载文字；虚线括号只表达胶囊组归属，不连接到节点正文。
- 若要素不是 3 个，优先改用中心扩散图或模块系统，不在此母题里新增第 4 个环绕节点。

### P6 分层核心诊断

适用场景：表达一组顶部输入要素汇入中心双核心，再对应左右两侧分层说明与底部问题诊断。适合“输入调研-现状/核心-左右分层分析-问题归纳”“外部信息输入 + 中心判断 + 两侧结构化解释”的诊断型结构。

- `bottomItems` 固定支持 3 个问题标签，每个最长 10 个中文字符。
- 顶部、中心、底部三组胶囊容器只表达归组关系；绿色桥接只表达上下输入输出关系，不承载文字。
- 若需要超过 3 行说明或超过 2 个核心节点，优先拆页或改用模块系统。

## 硬门禁

生成后运行逻辑图形质检。以下情况必须失败并返工：

- 图形、节点、旁注或分组越过画布、安全区、标题区或页脚区。
- 节点之间重叠，或不相关节点距离过近。
- 连接线、箭头、连接标签压住节点正文。
- 单页主焦点超过 1 个。
- 节点、旁注、徽标或连接标签超出容量上限。
- 同一层级组件的字号、线宽、圆角、颜色或间距 token 不一致。
